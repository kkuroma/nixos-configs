## The Core Stack: `nix flakes` + `direnv` + `nix-direnv`

The canonical answer to "how does VSCode know about my Nix shell" is **direnv**. You can use `nix-direnv` and the VSCode extension `direnv` for integration — `nix-direnv` hooks your shell's `cd` command and automatically activates the environment when you enter a directory.

`nix-direnv` is the critical companion piece. Without it, direnv would re-evaluate the flake on every shell start. `nix-direnv` creates a GC root pointing at the cached environment, so it survives `nix-collect-garbage` and reuses the cached result unless `flake.nix` or `flake.lock` actually changed.

The VSCode side: install the **`mkhl.direnv`** extension. It enables automatic Python interpreter selection for execution and debugging in VSCode by loading direnv's environment variables, including `$PYTHONPATH`, that VSCode will use to select the correct Python interpreter.

## The Division of Responsibility: Nix vs. UV

The key insight for your ML workflow is **don't fight Nix trying to manage every Python package**. The best practice is:

Nix provides the Python interpreter; `uv` manages the virtualenv and packages. This avoids the pain of Nix trying to manage every Python package (which often breaks with packages that have C extensions).

Your `flake.nix` devShell for ML would look roughly like:

```nix
devShells.ml = pkgs.mkShell {
  buildInputs = [
    pkgs.python312
    pkgs.uv
    pkgs.cudaPackages.cudatoolkit
    pkgs.cudaPackages.cudnn
    pkgs.gcc
    pkgs.pkg-config
  ];
  shellHook = ''
    # Auto-create and sync uv venv
    if [ ! -d ".venv" ]; then
      uv venv --python python3.12
    fi
    source .venv/bin/activate
    uv sync

    # CUDA env vars
    export CUDA_HOME=${pkgs.cudaPackages.cudatoolkit}
    export LD_LIBRARY_PATH=${pkgs.cudaPackages.cudatoolkit}/lib:$LD_LIBRARY_PATH
  '';
};
```

VSCode's Python extension will then find `.venv/bin/python` in the project root automatically. Point your editor to `.venv/bin/python` — most editors auto-detect it.

## Your `init-shell` Template System

Your idea of `init-shell --python --cuda` is essentially what the community calls **flake templates**. You can create generic `flake.nix` templates for commonly used languages in advance, which can be copied and modified as needed. The project environment takes the highest precedence (added to the front of `PATH`), and its dependencies will override those with the same name in the global environment.

Structure your template repo like:

```
~/nix-templates/
├── flake.nix           # registers templates
├── ml-cuda/
│   ├── flake.nix       # python + cuda + uv
│   └── .envrc          # use flake
├── frontend/
│   ├── flake.nix       # node + npx
│   └── .envrc
└── pentest/
    ├── flake.nix       # nmap, wireshark, metasploit etc.
    └── .envrc
```

Then `nix flake init -t github:yourname/nix-templates#ml-cuda` scaffolds the whole thing.

## The `.envrc` Pattern

Each project gets a `.envrc` file:

```bash
# For a pure Nix flake shell
use flake

# For Nix shell + uv venv auto-activation (hybrid)
use flake
layout python  # direnv's built-in python venv helper
```

Run `direnv allow` once to grant permission. After that, every `cd` into the directory activates the Nix shell silently, and leaving the directory deactivates it.

## Per-Domain Shell Summary

| Domain | Nix provides | Runtime manager | VSCode extension needs |
|---|---|---|---|
| ML/CUDA | `python`, `uv`, `cudatoolkit`, `gcc` | `uv` + `.venv` | Python extension → `.venv/bin/python` |
| Frontend | `nodejs`, `pnpm`/`npx` | `npm`/`pnpm` in project | ESLint, Prettier auto-detect via `direnv` env |
| Pentest/Kali stack | `nmap`, `wireshark`, `metasploit`, `ncat`, etc. | N/A (system tools) | Terminal inherits env |

## VSCode Settings to Lock In

In your project's `.vscode/settings.json`:

```json
{
  "python.defaultInterpreterPath": "${workspaceFolder}/.venv/bin/python",
  "python.terminal.activateEnvironment": false,
  "direnv.restart.automatic": true
}
```

Setting `activateEnvironment: false` is important — you don't want VSCode's Python extension fighting with direnv over activation. The `.venv` is already active from direnv's `shellHook`.

---

## The One Gotcha: Launching VSCode From The Shell

VSCode must be **launched from within the activated environment** to inherit it. Either:
- `cd project && direnv allow && code .` in your terminal, or
- Use the `mkhl.direnv` extension which re-sources `.envrc` inside VSCode's own process.

The `nix-env-selector` extension can also make switching between different nix-shell environments within VSCode easy for manually switching between coding projects. But `direnv-vscode` is the more automated and modern approach.
